4. Règle U_NoLot (numéro de lot)

Obligatoire — un BL sans U_NoLot conforme passe côté SAP mais est rejeté au contrôle Gervifrais. Règle confirmée par probe sur des BL réels.

Langage : Pseudo-code — règle métier

U_NoLot = "EM" + DocNum du DERNIER PurchaseDeliveryNote
          contenant l'ItemCode DANS le WarehouseCode de la ligne

# Cascade de fallback si rien trouvé :
#   1. (ItemCode + WarehouseCode)  → cas nominal
#   2. (ItemCode, dernière réception toutes confondues)
#   3. "EM0000" (constante GERVIFRAIS_LOT_DEFAUT)

# Exemples observés sur la base TEST :
#   FE1SL reçu en 000 via PDN n°22739  →  U_NoLot = "EM22739"
#   FE1SL reçu en 01  via PDN n°22755  →  U_NoLot = "EM22755"

Conséquence métier : un même ItemCode livré depuis 2 entrepôts différents = 2 lots différents = 2 lignes distinctes dans DocumentLines. Le split se fait dans l'input Orderlion, pas dans le builder de payload.

6. Code minimal — split par lot

Le split (un même article sur 2 entrepôts → 2 lignes) est dans l'input Orderlion. Le builder ne fait que mapper chaque ligne d'input vers une ligne SAP enrichie.

Langage : TypeScript — pré-calcul du lot (cache 10 min)

// Construit deux index à partir des 500 dernières réceptions :
//   byItemWhs : "ItemCode|Warehouse" → dernier DocNum vu (cas nominal)
//   byItem    : "ItemCode"           → dernier DocNum vu (fallback)
async function buildLotMaps() {
  const byItemWhs = new Map<string, number>();
  const byItem    = new Map<string, number>();

  for (let skip = 0; skip < 500; skip += 50) {
    const r = await sapGet(
      "PurchaseDeliveryNotes"
      + "?$top=50&$skip=" + skip
      + "&$orderby=DocNum desc"
      + "&$select=DocNum,DocumentLines"
    );
    if (!r.value?.length) break;

    for (const doc of r.value) {
      for (const line of doc.DocumentLines ?? []) {
        if (!line.ItemCode) continue;

        // On garde le DocNum max par sécurité (au cas où l'ordre flotte)
        const prevItem = byItem.get(line.ItemCode) ?? 0;
        if (doc.DocNum > prevItem) byItem.set(line.ItemCode, doc.DocNum);

        if (line.WarehouseCode) {
          const key  = line.ItemCode + "|" + line.WarehouseCode;
          const prev = byItemWhs.get(key) ?? 0;
          if (doc.DocNum > prev) byItemWhs.set(key, doc.DocNum);
        }
      }
    }
  }
  return { byItemWhs, byItem };
}

Langage : TypeScript — résolution du lot pour une ligne donnée

// Cascade : (item+entrepôt) → (item seul) → fallback constant.
function resolveLot(
  maps: { byItemWhs: Map<string, number>; byItem: Map<string, number> },
  itemCode: string,
  warehouseCode: string,
): string {
  // 1. Cas nominal : réception vue pour cet item dans cet entrepôt
  const exact = maps.byItemWhs.get(itemCode + "|" + warehouseCode);
  if (exact) return "EM" + exact;

  // 2. Fallback : item vu ailleurs → dernière réception toutes confondues
  const any = maps.byItem.get(itemCode);
  if (any) return "EM" + any;

  // 3. Fallback ultime (cf. GERVIFRAIS_LOT_DEFAUT)
  return "EM0000";
}

Langage : TypeScript — construction du payload final (avec split)

// Libellés humains des entrepôts (cf. §2)
const WHS_NAMES = {
  "000": "A/C - A/D",
  "01":  "Stock",
  "R1":  "J+1",
} as const;

// Une ligne d'input Orderlion = un couple (article, entrepôt)
type CartLine = {
  itemCode:      string;
  quantity:      number;                  // en unité de stock SAP (pie)
  warehouseCode: "000" | "01" | "R1";
};

async function buildOrderPayload(
  cart:     CartLine[],
  cardCode: string,                       // code client SAP (BusinessPartners.CardCode)
  dueDate:  string,                       // "YYYY-MM-DD"
  ref:      string,                       // référence Orderlion → NumAtCard
) {
  const lotMaps = await buildLotMaps();

  // Cache local des méta-articles : U_Pays, U_GER_Marque, U_GER_Det_Condt,
  // SalesQtyPerPackUnit. Lecture batch via Items?$filter=...
  const items = await loadItemMeta(cart.map(l => l.itemCode));

  const documentLines = cart.map((l) => {
    const meta = items.get(l.itemCode);
    return {
      ItemCode:      l.itemCode,
      Quantity:      l.quantity,
      WarehouseCode: l.warehouseCode,
      U_NomMag:      WHS_NAMES[l.warehouseCode],
      U_NoLot:       resolveLot(lotMaps, l.itemCode, l.warehouseCode),
      // Renommage Items → ligne : U_GER_Det_Condt devient U_GER_Condi
      ...(meta?.U_Pays          && { U_GER_Pays:   meta.U_Pays }),
      ...(meta?.U_GER_Marque    && { U_GER_Marque: meta.U_GER_Marque }),
      ...(meta?.U_GER_Det_Condt && { U_GER_Condi:  meta.U_GER_Det_Condt }),
      // TPF : voir §7 pour le calcul et la réconciliation post-création
    };
  });

  const today = new Date().toISOString().slice(0, 10);
  return {
    CardCode:   cardCode,
    DocDate:    today,
    DocDueDate: dueDate,
    TaxDate:    today,
    NumAtCard:  ref,
    Comments:   "Via Orderlion",
    DocumentLines: documentLines,        // N lignes, une par (item, entrepôt)
  };
}

7. Réconciliation TPF après création

Si le prix est laissé au tarif SAP, le HT (LineTotal) n'est connu qu'après la création de la commande. Il faut donc relire la commande puis PATCHer les TPF par ligne avec les vraies valeurs. Sans ça, INTERFEL est à 0 ou faux.

Langage : HTTP — séquence des appels

# 1) Relire la commande qu'on vient de créer
GET    /Orders({DocEntry})

# 2) PATCH ciblé des DocumentLineAdditionalExpenses par ligne
PATCH  /Orders({DocEntry})

Langage : TypeScript — recalcul et PATCH

// Taux lus depuis AdditionalExpenses (mis en cache 10 min)
//   AdditionalExpenses(ExpensCode=2).U_Taux  →  INTERFEL
//   AdditionalExpenses(ExpensCode=3).U_Taux  →  DDG
const ITFEL_RATE = 0.0021;   // 0,21 % du LineTotal SAP
const DDG_RATE   = 0.02;     // 0,02 € par colis

async function reconcileTpf(docEntry: number) {
  // 1. Relire : SAP a calculé le vrai LineTotal de chaque ligne
  const order = await sapGet("Orders(" + docEntry + ")");

  const patchLines = [];
  for (const line of order.DocumentLines) {
    const meta    = await getItemMeta(line.ItemCode);
    const packDiv = meta?.SalesQtyPerPackUnit > 1
                      ? meta.SalesQtyPerPackUnit
                      : 1;
    const nbColis = (line.Quantity ?? 0) / packDiv;

    // Valeurs correctes attendues
    const itfel = round2((line.LineTotal ?? 0) * ITFEL_RATE);
    const ddg   = round2(nbColis * DDG_RATE);

    // Valeurs actuellement posées sur la ligne
    const exps    = line.DocumentLineAdditionalExpenses ?? [];
    const curItfel = exps.find(e => e.ExpenseCode === 2)?.LineTotal ?? 0;
    const curDdg   = exps.find(e => e.ExpenseCode === 3)?.LineTotal ?? 0;

    // Si écart > 1 centime → PATCH cette ligne
    if (Math.abs(itfel - curItfel) > 0.005
        || Math.abs(ddg - curDdg) > 0.005) {
      const merged = [];
      if (itfel > 0) merged.push(
        { GroupCode: 1, ExpenseCode: 2, LineTotal: itfel });
      if (ddg > 0) merged.push(
        { GroupCode: 2, ExpenseCode: 3, LineTotal: ddg });
      patchLines.push({
        LineNum: line.LineNum,
        DocumentLineAdditionalExpenses: merged,
      });
    }
  }

  if (patchLines.length > 0) {
    await sapPatch("Orders(" + docEntry + ")",
                   { DocumentLines: patchLines });
  }
}

const round2 = (n: number) => Math.round(n * 100) / 100;

8. Pièges connus

  • Typo SAP ExpensCode vs ExpenseCode : le master AdditionalExpenses expose ExpensCode (sans 'e'), alors qu'à la ligne le champ est ExpenseCode (avec). Les deux orthographes sont nécessaires.
  • Renommage U_GER_Det_Condt → U_GER_Condi : le champ master article et le champ ligne n'ont pas le même nom. À mapper explicitement.
  • U_NoLot manquant ou "EM0000" : passe côté SAP mais sera bloqué au contrôle Gervifrais. Logger un warning quand le fallback est utilisé.
  • Session SAP : idle > 30 min → 401 Unauthorized. Prévoir re-login automatique avec un verrou pour éviter les logins parallèles concurrents (sinon n logins simultanés = saturation).
  • TLS auto-signé (base TEST) : certif non chaîné. Côté Orderlion : accepter le cert (pinning) ou désactiver la vérif TLS sur cette base uniquement.
  • WarehouseCode hors {000, 01, R1} : SAP l'accepte mais l'article sort du flux télévente standard. Refuser en validation Orderlion.
  • Pré-validation conseillée avant POST : GET Items('XXX') pour chaque ItemCode unique + GET BusinessPartners('YYY'). Évite l'erreur SAP -2028 (article inexistant) et permet un message d'erreur clair côté Orderlion.
  • Quantités : SAP attend la quantité en unité d'inventaire (InventoryUOM, généralement pie). Si Orderlion travaille en colis, convertir avant POST.
